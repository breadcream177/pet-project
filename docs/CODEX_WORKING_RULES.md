# Codex 작업 원칙

이 문서는 챙겨펫 프로젝트에서 Codex와 함께 작업할 때 지킬 기준입니다.
목표는 코드를 많이 만드는 것이 아니라, 실제로 실행되고 유지보수 가능한 서비스를 작은 단위로 쌓아가는 것입니다.

## 1. 작업 목적

- 실제 사용자가 이해하기 쉬운 기능을 우선합니다.
- 혼자서 유지보수할 수 있는 구조를 유지합니다.
- 배포와 확장을 고려합니다.
- 이력서와 면접에서 설명 가능한 구현을 지향합니다.

## 2. 작업 방식

작업은 다음 순서로 진행합니다.

1. 현재 파일 구조와 관련 코드를 확인합니다.
2. 수정해야 할 파일 목록을 먼저 정리합니다.
3. 어떤 방식으로 고칠지 짧게 계획합니다.
4. 필요한 코드만 최소 단위로 수정합니다.
5. 수정 후 실행 방법, 테스트 방법, 변경 내용을 정리합니다.

큰 기능은 한 번에 만들지 않고 작은 작업 단위로 나눕니다.

## 3. 코드 구조 원칙

- 파일은 기능별로 분리합니다.
- 한 파일에 모든 코드를 몰아넣지 않습니다.
- 페이지, 컴포넌트, API 호출, 타입, 스타일 역할을 가능한 구분합니다.
- 공통 컴포넌트는 공통 폴더에 두고, 기능별 코드는 각 기능 폴더에 둡니다.

## 4. 기술 스택 기준

현재 챙겨펫의 구현 기준은 다음과 같습니다.

- Web/App UI: Next.js, React, TypeScript
- Mobile wrapper: Capacitor Android
- Auth/Database: Supabase
- Push: Firebase Cloud Messaging

새로운 라이브러리나 외부 서비스를 추가하기 전에는 이유와 영향 범위를 먼저 설명합니다.

## 5. 구현 원칙

- 초보 개발자가 읽어도 흐름을 이해할 수 있게 작성합니다.
- 중복은 줄이되 과도한 추상화는 피합니다.
- 에러 처리를 무시하지 않습니다.
- API 요청과 응답 흐름을 명확히 합니다.
- DB 구조는 나중에 확장 가능한 형태로 유지합니다.
- 사용자 입력값 검증을 고려합니다.
- 보안이 필요한 부분은 임시 구현과 실제 적용 가능한 구현을 구분합니다.

## 6. 금지 사항

- 전체 구조를 마음대로 갈아엎지 않습니다.
- 기존 정상 작동 기능을 이유 없이 변경하지 않습니다.
- 설명 없이 라이브러리를 추가하지 않습니다.
- 보안 키, API 키, 비밀번호를 코드에 직접 넣지 않습니다.
- 테스트하지 않은 코드를 완성된 것처럼 말하지 않습니다.
- 근거 없이 "아마 될 것 같다"고 확정하지 않습니다.

## 7. 변경 보고 방식

수정이 끝나면 다음을 정리합니다.

- 변경한 내용
- 수정한 파일
- 실행 방법
- 확인해야 할 부분
- 주의할 점

## 8. 질문 기준

다음 경우에는 작업 전에 사용자에게 확인합니다.

- DB 구조가 크게 바뀌는 경우
- 로그인, 보안, 권한 구조를 바꾸는 경우
- 기존 기능을 삭제해야 하는 경우
- 새로운 외부 API나 유료 서비스를 추가해야 하는 경우
- 프로젝트 방향 자체가 바뀌는 경우

## 9. 설명 기준

중요한 코드는 다음 관점으로 짧게 설명합니다.

- 이 코드가 왜 필요한지
- 어디에서 호출되는지
- 데이터가 어떤 흐름으로 이동하는지
- 에러가 나면 어디를 확인해야 하는지

## 10. 최종 목표

챙겨펫은 그럴듯한 코드 묶음이 아니라 실제로 실행되고 배포 가능한 서비스를 목표로 합니다.
모든 작업은 기능 완성, 유지보수성, 설명 가능성, 배포 가능성을 기준으로 판단합니다.
